Formal Requirements Specification

Course and Session Duplication · Enterprise and client-facing approach

This is a portfolio adaptation of the same product scenario, created to demonstrate a more formal enterprise and client-facing documentation approach. It does not represent a separate client engagement.

Company names, internal identifiers, endpoints, service names, and unverifiable metrics have been removed or generalised. Requirement codes have been renumbered neutrally.

§1Business context

1.1 Business need

Administrators managing recurring scheduled training programmes are currently required to recreate every new course from scratch, including all associated sessions. The process is time-consuming, error-prone, and does not scale for organisations running a high volume of programmes. The duplication capability addresses a direct and concrete operational need from administrative users.

1.2 Business objectives

  1. Reduce the time required to configure new courses
  2. Decrease manual errors when creating courses with recurring structure
  3. Increase configuration consistency across programme editions
  4. Improve overall administrative satisfaction and operational efficiency

Objectives as defined at specification time. No measured outcomes are presented.

§2Scope definition

2.1 System scope

The system in scope is the scheduled-training module of the platform, specifically course and session management. Duplication applies exclusively to the applicable course type and produces a new course fully independent from the source.

2.2 Scope boundaries

Out of scope: enrolments, attendance records, historical completions, and all learner-related data.

§4Glossary

§7Functional requirements

Requirements use the normative verbs SHALL (mandatory), SHOULD (recommended), and MAY (optional).

FR-01Initiate the duplication flow
Statement
The system SHALL allow an authorised administrator to initiate the duplication flow for an existing course.
Rationale
Reduce manual course configuration time (business objective 1)
Priority
High
Fit criterion
An administrator with the relevant permissions can select "Duplicate" from a course action menu and the system opens the duplication stepper.
Dependencies
BR-03 · BR-04 · CON-01
FR-02Select duplication type
Statement
The system SHALL allow the administrator to select among three duplication modes: course only, course with all sessions, or course with selected sessions.
Rationale
Provides granular control over what is duplicated based on programme needs
Priority
High
Fit criterion
In step 1, the administrator selects one of three options; the selection determines the behaviour of subsequent steps.
Dependencies
FR-01 · BR-01 · CON-03
FR-03Select individual sessions
Statement
The system SHALL allow the administrator to select a subset of sessions to duplicate when the subset mode is active.
Rationale
Operational flexibility: not all sessions of a course may be relevant for the new instance
Priority
High
Fit criterion
The session list is displayed with individual selection controls; only selected sessions are included in the duplication.
Dependencies
FR-02 · BR-01 · BR-04
FR-04Validate data before confirmation
Statement
The system SHALL validate duplication data before allowing final confirmation.
Rationale
Prevent creation of courses or sessions with inconsistent or invalid data
Priority
High
Fit criterion
If validation fails, the system shows field-level errors and blocks progression to the review step; no data is created.
Dependencies
FR-02 · FR-03 · CON-03
FR-05Review step and confirmation
Statement
The system SHALL present a summary of what will be duplicated before executing the operation.
Rationale
Ensures full administrator awareness before an irreversible action is taken (CON-03)
Priority
High
Fit criterion
The review step shows the new course name and the list of included sessions; the administrator can confirm or navigate back.
Dependencies
FR-04 · BR-02 · CON-03
FR-06Handle partial duplication
Statement
The system SHALL complete course creation even if one or more sessions fail to duplicate, notifying the administrator of the sessions not copied.
Rationale
BR-05: avoids total loss of work in case of partial session failures
Priority
High
Fit criterion
On partial failure, the course is created and a warning lists the sessions not duplicated; no failed session is silently ignored.
Dependencies
BR-05 · FR-05

§9Use case model

UC-01Duplicate course
Goal
Create a duplicated course from an existing one, with or without associated sessions
Primary actor
Administrator
Preconditions
The course exists; the user holds management permissions; the course is of the applicable type
Trigger
Administrator selects "Duplicate" from the course action menu
Main scenario
1. Administrator selects "Duplicate" · 2. System opens the modal stepper · 3. Administrator selects duplication type · 4. If subset mode: administrator selects sessions · 5. System validates the selection · 6. System presents the review summary · 7. Administrator confirms · 8. System creates the new course with the selected sessions · 9. System shows a success notification linking to the new course
Alternative flows
AF-01 Course only: step 4 is skipped; only metadata is duplicated. AF-02 All sessions: step 4 is skipped; all sessions are included automatically.
Exception flows
EF-01 Permission denied: operation blocked; no data created. EF-02 Validation failure: field-level errors shown; progression blocked. EF-03 Source deleted mid-flow: hard error; modal closed; no data created. EF-04 Partial failure: course created; warning lists failed sessions.
Postconditions
Success: a new independent course exists with the selected sessions; the source is unchanged. Failure: no new course is created; the source is unchanged.

§11System modelling

The specification included three complementary models describing the capability from different angles.

  • State model — the lifecycle of a course during duplication, from idle through selection, review, and execution to either a completed or failed end state.
  • Activity model — the full flow with branching by duplication type, permission checks, validation gates, and the three possible outcomes: success, partial success, and error.
  • Sequence model — the interaction between the administrator, the interface, and the platform components handling authorisation, course creation, and session creation.

Diagrams are omitted from this published version, as they expose internal component structure.

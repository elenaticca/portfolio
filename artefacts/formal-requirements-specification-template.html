§6Requirements conventions and lifecycle

Establishes the coding, prioritisation, and lifecycle conventions used throughout the document, ensuring consistency and supporting traceability.

6.1 Requirement types

TypePrefixDescription
Functional requirementFRA behaviour the system must support
Non-functional requirementNFRA quality attribute or constraint on how the system behaves
Business ruleBRA policy, logic, or constraint derived from business context
ConstraintCONAn external or organisational limitation on the system or delivery

6.2 Priority scale

LabelMeaning
HIGHRequired for the system to meet its core objectives — must be delivered
MEDIUMStrongly recommended — should be delivered in this iteration unless constrained
LOWDesirable — can be deferred without blocking core delivery

6.3 Normative language

Requirements use the following normative verbs as defined in RFC 2119.

VerbMeaning
SHALLMandatory — the system must implement this behaviour
SHOULDStrongly recommended — deviation requires documented justification
MAYOptional — implementation is permitted but not required

6.4 Stability and status lifecycle

StatusMeaning
DraftUnder preparation — not yet reviewed or baselined
Under reviewCirculated for stakeholder review
ApprovedAgreed and baselined — changes require formal change control
DeferredExcluded from current scope — candidate for a future iteration
RejectedFormally excluded with documented rationale

§10Use cases and scenarios

10.1 Use case model

UC-01[Use case title]
Goal
What the primary actor is trying to achieve through this interaction
Primary actor
The main user role or system initiating the interaction
Secondary actors
Other systems or roles involved in fulfilling the use case
Preconditions
Conditions that must be true before the use case can begin
Trigger
What initiates the use case — an actor action, a scheduled event, an external message
Main success scenario
Numbered steps from the initial action through to the confirmed outcome state
Alternative flows
AF-01, AF-02 — alternative paths that also lead to a successful outcome
Exception flows
EF-01, EF-02 — what happens when something goes wrong: permission denied, validation failure, external system unavailable
Postconditions
System state after success, and after failure with no unintended data changes

10.2 Scenarios

IDGivenWhenThen
S-01[Context — who the actor is and what state the system is in][The specific action the actor takes][The expected outcome]

Scenarios should collectively cover the main success path, significant alternative paths, and key exception flows. Each relates back to specific requirements, rules, and use cases.

§11–§12Data, interfaces, models and diagrams (optional)

§11 Data and interface requirements — included when data exchange or system integration is relevant. Covers systems involved, data exchanged, interface contracts, data ownership, retention and deletion, access rules, and known technical constraints.

§12 Models and diagrams — context diagram, process or activity diagram, state model, sequence diagram, and optionally a data model. Diagrams are supplementary to the written requirements, not a replacement for them.

§16Approval and change control

16.1 Document approval

The document is baselined when all required approvers have confirmed in writing or via the project's formal approval process. Approvers are recorded with name, role, organisation, and approval date.

16.2 Change control

After baselining, any change must follow the agreed change control process. Approved changes are recorded in the revision history with a change request ID, description, requester, approver, date, and resulting version.

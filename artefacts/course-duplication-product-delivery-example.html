Product Requirements — Delivery Specification

Course and Session Duplication · Applied product approach

An anonymised example based on professional product work. Company names, internal identifiers, endpoints, service names, telemetry, and implementation details have been removed or generalised. Requirement codes have been renumbered neutrally.

01Overview

Goal

Allow platform administrators to duplicate a scheduled training course, optionally including its associated sessions — all of them or a selected subset — removing the need to reconfigure recurring programmes by hand.

Problem statement

When an administrator needs a new course similar to an existing one, they must recreate all settings and sessions from scratch. No duplication capability exists for this course type, unlike other course types already supported by the platform. The result is operational inefficiency, error risk, and wasted configuration time.

Business value

  • Reduced course setup time for administrators
  • Fewer manual configuration errors on recurring programmes
  • Greater consistency across programme editions
  • Improved overall administrative efficiency

Stated as expected value at specification time. No measured post-release results are presented.

02Scope

In scope

  • Duplicate course metadata and settings
  • Duplicate all associated sessions
  • Duplicate a user-selected subset of sessions
  • Multi-step selection flow: type → sessions → review
  • Data validation before confirmation
  • Review step summarising what will be duplicated

Out of scope

  • Learner enrolment duplication
  • Attendance record duplication
  • Historical completion data duplication
  • Session structure redesign
  • Duplication of other course types

04Assumptions and constraints

Assumptions

  • The existing permission model remains unchanged
  • Session structure — dates, location, instructor — is already defined on the source course
  • The capability applies exclusively to the applicable course type
  • The duplicated course is fully independent from the original once created

Constraints

TypeDescription
BusinessSource course must remain unchanged. Enrolments and historical learner data must not be copied under any circumstances.
TechnicalExisting interfaces must be reused where possible. The existing permission model must be respected without architectural changes.
UXUsers must clearly understand what will be duplicated before confirming. A review step is mandatory.

05User flow

The duplication flow is a three-step modal stepper.

Step 01

Select type

Course only · All sessions · Selected sessions

Step 02

Select sessions

Active only when "selected sessions" is chosen

Step 03

Review and confirm

Summary of what will be duplicated; confirm or go back

06Functional requirements

FR-01Initiate duplication flow
Description
The administrator can initiate the duplication flow directly from a course page.
UI behaviour
A "Duplicate" action is available in the course actions menu, visible only for the applicable course type. Selecting it opens the modal stepper.
Validation
User must hold the relevant management permission. Course must exist and not be in a deleted state.
Edge cases
Course deleted during the flow → show error and close the stepper. User loses permission mid-flow → block with an access denied message.
Acceptance
Given an administrator with management permissions is viewing a course · When they select "Duplicate" from the actions menu · Then the system opens the duplication stepper
FR-02Select duplication type
Description
In step 1, the administrator selects among three options: course only · course with all sessions · course with selected sessions.
UI behaviour
Radio button group with three options. Selection state is reflected in the stepper header. "Next" is enabled only when one option is selected.
Validation
Exactly one option must be selected to proceed.
Edge cases
User navigates back and changes selection → prior state must be maintained or coherently reset.
Acceptance
Given the administrator is in step 1 · When they select "course with selected sessions" · Then the system enables step 2 with the session selection list
FR-03Select individual sessions
Description
When the subset mode is active, step 2 displays all sessions of the source course with individual checkboxes.
UI behaviour
Checkbox list showing name, date, location, and instructor per session, with a select all / deselect all control. "Next" is enabled only when at least one session is selected.
Validation
At least one session must be selected. Unavailable sessions are shown as disabled.
Edge cases
No sessions available → empty state with an explanatory message. Session becomes unavailable during selection → exclude it and notify inline.
Acceptance
Given the administrator is in step 2 and the course has available sessions · When they select a subset · Then only the selected sessions are included
FR-04Review and confirm
Description
Before creation, the administrator sees a summary of what will be duplicated and can confirm or go back.
UI behaviour
Review screen showing the new course name, the list of included sessions, and confirm and back controls.
Validation
The system runs a final validation pass before executing creation. Errors are shown inline.
Edge cases
Validation failure → show specific errors and block confirmation. Session timeout → warn the user; no partial data is persisted.
Acceptance
Given the administrator has completed selection and is in the review step · When they confirm · Then the system creates the duplicated course with the selected sessions and shows a success notification linking to it

07Business rules

08UX states

StateBehaviour

ErrorError banner explaining that duplication could not be completed, with detail where available
Partial warningWarning banner confirming the course was created while listing sessions that were not copied
Empty stateMessage explaining that the course has no sessions available for duplication
SuccessConfirmation notification with a link to the newly created course

09Error handling

ScenarioExpected behaviour
Permission deniedBlock the action and explain that the user lacks permission to duplicate the course
Partial duplicationCourse is still created; a warning lists the sessions that were not duplicated
Session unavailable during selectionSession is excluded from the list; inline notification shown
Source course deleted during flowHard error: close the modal and show an error on the page; no data created
Timeout or generic errorGeneric error message shown; no partial data is persisted

10QA considerations

  • Permissions: with permission, without permission, and permission revoked mid-flow
  • Duplication type: course only, all sessions, selected sessions
  • Session selection: partial, full, and none attempted
  • Validation: valid data, invalid data, unavailable sessions
  • Partial failures: one or more sessions failing during duplication
  • Concurrency: source course deleted while the flow is open
  • UI states: loading, error, success, empty, partial warning
  • Data integrity: source course must not be modified at any point

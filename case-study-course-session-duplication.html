Case-01

Course and Session Duplication

An anonymised B2B SaaS product case

This case study is based on anonymised professional product work. Company-specific information, internal identifiers, and implementation details have been removed or generalised.

01Case overview

A B2B SaaS learning platform supported scheduled training courses made up of individual sessions, each with its own date, location, and assigned trainer. Organisations running recurring programmes had to rebuild every new edition manually, course settings and sessions alike.

The work covered the analysis and specification of a duplication capability: defining what could be copied, what had to be excluded, who was allowed to perform the action, how the system should behave at each step, and what should happen when something went wrong partway through.

02Business problem

Administrators creating a new course similar to an existing one had to recreate all settings and every session from scratch. No duplication capability existed for this course type, although comparable mechanisms already existed elsewhere on the platform.

The consequences were operational rather than technical: configuration time that scaled linearly with the number of sessions, inconsistency between editions of the same programme, and a rising chance of manual error as programme volume grew.

Expected value

  • Reduced setup time when configuring recurring course editions
  • Fewer manual configuration errors on repeated programme structures
  • Greater consistency across editions of the same programme
  • Improved day-to-day operational efficiency for administrative users

These were the anticipated benefits used to justify the work. They are stated as expected value, not as measured results.

03My role and contribution

I led the requirements engineering and functional analysis for this capability, working between product management, design, engineering, and QA.

  • Clarified the business need and translated it into a defined, bounded scope
  • Established which data could be copied and which had to be excluded, and why
  • Specified functional and non-functional requirements with verifiable criteria
  • Modelled the user flow, system behaviour, and permission constraints
  • Documented edge cases, error states, and partial-failure handling
  • Defined acceptance criteria to support development and testing
  • Surfaced open decisions that required product input before implementation

04Analysis approach

The central difficulty was not building a copy function. It was deciding precisely what a copy should mean in a system where course data and learner data are entangled.

Duplicating a course naively would carry across learner registrations and historical records, producing a new course populated with people who never enrolled in it. The analysis therefore started from the exclusions and worked outwards.

  1. Separated course configuration from learner-generated data, and drew the boundary between them
  2. Mapped the administrator's journey as a stepped flow with an explicit review point before any irreversible action
  3. Defined business rules covering data integrity, permissions, and the applicable course type
  4. Specified system behaviour for each branch of the flow, including failure paths
  5. Wrote acceptance criteria in a form testable by QA without further interpretation

05Scope and explicit exclusions

Stating the exclusions as firmly as the inclusions was the decision that shaped everything downstream.

In scope

  • Course configuration and settings
  • All associated sessions
  • A user-selected subset of sessions
  • Stepped selection flow with review
  • Validation before confirmation

Explicitly excluded

  • Learner enrolments
  • Attendance records
  • Historical completion data
  • Redesign of session structure
  • Other course types

The duplicated course is fully independent from its source once created. The source itself is never altered by the operation.

06Simplified user and system flow

Step 01

Select type

Course only, course with all sessions, or course with selected sessions.

Step 02

Select sessions

Active only when a subset is chosen. Unavailable sessions are shown but cannot be selected.

Step 03

Review and confirm

Summary of exactly what will be created. The administrator can confirm or go back.

Behind the flow, the system verifies permissions before the first step, validates the selection before the review, and only then creates the new course and its sessions.

07Selected business rules

IDRuleRationale
BR-01Learner registrations are never duplicatedPrevents irrelevant historical data and inconsistent registrations on the new course
BR-02Source course data remains unchanged after duplicationData integrity and no regression for learners already enrolled
BR-03Only users with the relevant management permission may initiate duplicationAccess control and data security
BR-04The capability applies only to the applicable course typeScope boundary; other course types have separate mechanisms
BR-05If some sessions fail, the course is still createdPrevents total loss of work; failures are surfaced rather than hidden

08Selected functional requirements

IDRequirement
FR-01An authorised administrator can initiate the duplication flow from an existing course
FR-02The administrator can choose among three duplication modes
FR-03The administrator can select a subset of sessions when that mode is active
FR-04The system validates the selection before allowing final confirmation
FR-05The system presents a review summary before executing the operation
FR-06The system completes course creation even when individual sessions fail, and reports which ones

Non-functional requirements covered responsiveness under normal load, recoverability from partial failures, server-side permission enforcement, and verified immutability of the source course.

09Acceptance criteria

Each requirement was paired with criteria written so QA could verify them without needing to reinterpret intent.

GivenWhenThen
An authorised administrator is viewing a courseThey duplicate it with all sessionsA new course is created with all sessions; the source is unchanged
The administrator chooses a subset of sessionsThey select some and confirmOnly the selected sessions appear in the new course
The administrator chooses course onlyThey confirm in the review stepA new course is created with no sessions
A user lacks the required permissionThey attempt to initiate duplicationThe action is blocked; no data is created
One session fails during creationThe system processes all sessionsThe course is created; a warning identifies the failed session

10Edge cases and partial-failure considerations

Most of the analytical effort went here. The straightforward path was quick to specify; the exceptions determined whether the feature would be trustworthy.

  • Partial failure. If some sessions fail while others succeed, the course is still created and the administrator is told exactly which sessions were not copied. Silent failure was explicitly rejected.
  • Permission lost mid-flow. The operation is blocked rather than half-completed.
  • Source deleted while the flow is open. The flow terminates with a clear error and no data is created.
  • Session becomes unavailable during selection. It is excluded and the administrator is notified inline rather than at confirmation.
  • Validation failure. Field-level errors are shown and progression is blocked before anything is created.
  • No sessions available. An explanatory empty state, rather than an empty list with no context.

Open decisions raised

Three questions were surfaced for product decision before development rather than resolved by assumption: whether archived sessions should be selectable, whether the new course name should be editable or auto-assigned, and what state a course should be in following a partial failure.

11Two documentation approaches

The same scenario is documented twice, to show how requirements structure changes with delivery context rather than with content.

The product delivery specification is the applied artefact: requirement descriptions paired with interface behaviour, validation rules, and edge cases, written for a cross-functional team working in iterative cycles.

The formal requirements specification restructures the identical scenario for an enterprise or client-facing setting: normative statements with rationale and fit criteria, a full use case model, and traceability across rules, scenarios, and acceptance criteria.

The formal version is a portfolio adaptation. It does not represent a separate client engagement.
